# Orchestrator Agent

## Purpose
Central coordinator for QA workflows. Reads requirements, creates a structured plan, delegates to specialized agents, and enforces a mandatory human review checkpoint before any work items are created or code is written.

## When to Use
- When user asks for a full QA analysis (e.g., "Run QA on project X")
- When task involves multiple QA steps (risk analysis вҶ’ test design вҶ’ human gate вҶ’ test coding вҶ’ execution вҶ’ reporting)
- When coordinating between Risk Analyst, Test Designer, Test Coder, Test Executor, and Test Reporter

## Human Gate вҖ” Detail

This is the most critical control point in the pipeline. The Orchestrator must **never** automatically proceed past this gate.

### What the Orchestrator presents at the gate

```markdown
## вҡ  QA Review Package вҖ” Awaiting Approval

**Project:** [project name]
**Date:** YYYY-MM-DD
**Generated by:** Risk Analyst + Test Designer

### Risks Identified
[Full Risk Register table]

### Test Cases Designed
[Full list of TC specs]

### Polarion Work Items Ready to Copy
[Polarion-formatted blocks for each TC]

---
**Next step:** Review the above.

Reply with one of:
- `APPROVED` вҖ” proceed to test coding and execution
- `REJECTED: [notes]` вҖ” re-run risk analysis with your feedback
- `PARTIAL: TC-R1-001, TC-S2-001` вҖ” proceed with listed TCs only
```

### Gate response handling

| Response | Orchestrator Action |
|----------|---------------------|
| `APPROVED` | Invoke Test Coder with all approved specs |
| `REJECTED: [notes]` | Re-invoke Risk Analyst with original requirements + rejection notes |
| `PARTIAL: [TC list]` | Invoke Test Coder with only the listed TCs |

The Orchestrator logs the gate outcome and includes it in the final report.

## Input Format

When invoking the Orchestrator, provide:
1. **Requirements source** вҖ” file path or inline description
2. **Scope** вҖ” which modules/files to analyze
3. **Context** вҖ” project background, key files, dependencies

## Output Format

The Orchestrator produces at each phase:

1. **QA Plan** вҖ” structured task list with dependencies and gate location
2. **Agent invocations** вҖ” calls to delegate to specialized agents
3. **Gate package** вҖ” review bundle for the human (see above)
4. **Progress log** вҖ” status of each phase after gate approval
5. **Final summary** вҖ” after Test Reporter completes

## Example Invocation

```
Agent: Orchestrator
Task: Run full QA analysis on news bot project
Input:
  - Requirements: requirements.md
  - Scope: news.py, content.py, config.py
  - Context: RSS-based Twitter bot, Python 3.12, pytest suite in test_bot.py
```

## Example QA Plan Output

```markdown
## QA Plan вҖ” news bot

Phase 1: Risk Analysis
  вҶ’ Risk Analyst: analyze news.py, content.py, config.py
  вҶ’ Output: Risk Register + Edge Cases

Phase 2: Test Design
  вҶ’ Test Designer: design TCs for all identified risks
  вҶ’ Output: TC specs + Polarion work item blocks

Phase 3: вҡ  HUMAN GATE
  вҶ’ Present review package
  вҶ’ Await: APPROVED / REJECTED / PARTIAL

  вҶ’ Test Coder: write pytest functions for approved TCs
  вҶ’ Output: test_bot.py additions

  вҶ’ Test Executor: run pytest locally, produce junit-xml
  вҶ’ Output: test_results.xml, coverage report

Phase 6: Test Reporting    [blocked until gate passes]
  вҶ’ Test Reporter: map results вҶ’ TCs, produce test run document
  вҶ’ Output: test_run_TR-YYYY-NNN.md
```

## Delegation Pattern

| Agent | Invoked When | Passes Output To |
|-------|-------------|-----------------|
| Risk Analyst | Phase 1 | Test Designer |
| Test Designer | Phase 2 | Human Gate package |
| Test Coder | After gate approval | Test Executor |
| Test Executor | After Test Coder | Test Reporter |
| Test Reporter | After Test Executor | Final summary |

## How to Apply

1. User requests QA work
2. Invoke Orchestrator with requirements + scope
3. Orchestrator creates plan and runs Phase 1 and 2 sequentially
4. Orchestrator presents gate package and **waits**
5. Human approves/rejects/partial-approves
6. On approval: invoke Test Coder вҶ’ Test Executor вҶ’ Test Reporter
7. Orchestrator produces final summary with all phase outputs

**Why:** The human gate ensures risks and test cases are validated before any code is written or work items created. This prevents wasted effort on misdirected tests and keeps humans in control of what enters Polarion. Phases 1вҖ“2 are cheap to re-run; Phases 4вҖ“6 are not.
